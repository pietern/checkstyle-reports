14  /**
15   * A {@link Stream} that has additional operations to monitor or control behavior when used from a {@link Cache}.  Note that
16   * you may only use these additional methods on the CacheStream before any intermediate operations are performed as
17   * a {@link Stream} is returned from those methods.
18   *
19   * <p>Whenever the iterator or spliterator methods are used the user <b>must</b> close the {@link Stream}
20   * that the method was invoked on after completion of its operation.  Failure to do so may cause a thread leakage if
21   * the iterator or spliterator are not fully consumed.</p>
22   *
23   * <p>When using stream that is backed by a distributed cache these operations will be performed using remote
24   * distribution controlled by the segments that each key maps to.  All intermediate operations are lazy, even the
25   * special cases described in later paragraphs and are not evaluated until a final terminal operation is invoked on
26   * the stream.  Essentially each set of intermediate operations is shipped to each remote node where they are applied
27   * to a local stream there and finally the terminal operation is completed.  If this stream is parallel the processing
28   * on remote nodes is also done using a parallel stream.</p>
29   *
30   * <p>Parallel distribution is enabled by default for all operations except for {@link CacheStream#iterator()} &
31   * {@link CacheStream#spliterator()}.  Please see {@link CacheStream#sequentialDistribution()} and
32   * {@link CacheStream#parallelDistribution()}.  With this disabled only a single node will process the operation
33   * at a time (includes locally).</p>
34   *
35   * <p>Rehash aware is enabled by default for all operations.  Any intermediate or terminal operation may be invoked
36   * multiple times during a rehash and thus you should ensure the are idempotent.  This can be problematic for
37   * {@link CacheStream#forEach(Consumer)} as it may be difficult to implement with such requirements, please see it for
38   * more information.  If you wish to disable rehash aware operations you can disable them by calling
39   * {@link CacheStream#disableRehashAware()} which should provide better performance for some operations.  The
40   * performance is most affected for the key aware operations {@link CacheStream#iterator()},
41   * {@link CacheStream#spliterator()}, {@link CacheStream#forEach(Consumer)}.  Disabling rehash can cause
42   * incorrect results if the terminal operation is invoked and a rehash occurs before the operation completes.  If
43   * incorrect results do occur it is guaranteed that it will only be that entries were missed and no entries are
44   * duplicated.</p>
45   *
46   * <p>Any stateful intermediate operation requires pulling all information up to that point local to operate properly.
47   * Each of these methods may have slightly different behavior, so make sure you check the method you are utilizing.</p>
48   *
49   * <p>An example of such an operation is using distinct intermediate operation. What will happen
50   * is upon calling the terminal operation a remote retrieval operation will be ran using all of
51   * the intermediate operations up to the distinct operation remotely.  This retrieval is then used to fuel a local
52   * stream where all of the remaining intermediate operations are performed and then finally the terminal operation is
53   * applied as normal.  Note in this case the intermediate iterator still obeys the
54   * {@link CacheStream#distributedBatchSize(int)} setting irrespective of the terminal operator.</p>
55   *
56   * @param <R> The type of the stream
57   * @since 8.0
58   */
59  public interface CacheStream<R> extends Stream<R> {
60     /**
61      * This would disable sending requests to all other remote nodes compared to one at a time. This can reduce memory
62      * pressure on the originator node at the cost of performance.
63      * <p>Parallel distribution is enabled by default except for {@link CacheStream#iterator()} &
64      * {@link CacheStream#spliterator()}</p>
65      * @return a stream with parallel distribution disabled
66      */
67     CacheStream<R> sequentialDistribution();
68  
69     /**
70      * This would enable sending requests to all other remote nodes when a terminal operator is performed.  This
71      * requires additional overhead as it must process results concurrently from various nodes, but should perform
72      * faster in the majority of cases.
73      * <p>Parallel distribution is enabled by default except for {@link CacheStream#iterator()} &
74      * {@link CacheStream#spliterator()}</p>
75      * @return a stream with parallel distribution enabled.
76      */
77     CacheStream<R> parallelDistribution();
78  
79     /**
80      * Filters which entries are returned by what segment they are present in.  This method can be substantially more
81      * efficient than using a regular {@link CacheStream#filter(Predicate)} method as this can control what nodes are
82      * asked for data and what entries are read from the underlying CacheStore if present.
83      * @param segments The segments to use for this stream operation.  Any segments not in this set will be ignored.
84      * @return a stream with the segments filtered.
85      */
86     CacheStream<R> filterKeySegments(Set<Integer> segments);
87  
88     /**
89      * Filters which entries are returned by only returning ones that map to the given key.  This method will <b>always</b>
90      * be faster than a regular {@link CacheStream#filter(Predicate)} if any keys must be retrieved remotely or if a
91      * cache store is in use.
92      * @param keys The keys that this stream will only operate on.
93      * @return a stream with the keys filtered.
94      */
95     CacheStream<R> filterKeys(Set<?> keys);
96  
97     /**
98      * Controls how many keys are returned from a remote node when using a stream terminal operation with a distributed
99      * cache to back this stream.  This value is ignored when terminal operators that don't track keys are used.  Key
100     * tracking terminal operators are {@link CacheStream#iterator()}, {@link CacheStream#spliterator()},
101     * {@link CacheStream#forEach(Consumer)}.  Please see those methods for additional information on how this value
102     * may affect them.
103     * <p>This value may be used in the case of a a terminal operator that doesn't track keys if an intermediate
104     * operation is performed that requires bringing keys locally to do computations.  Examples of such intermediate
105     * operations are {@link CacheStream#sorted()}, {@link CacheStream#sorted(Comparator)},
106     * {@link CacheStream#distinct()}, {@link CacheStream#limit(long)}, {@link CacheStream#skip(long)}</p>
107     * <p>This value is <b>always</b> ignored when this stream is backed by a cache that is not distributed as all
108     * values are already local.</p>
109     * @param batchSize The size of each batch.  This defaults to the state transfer chunk size.
110     * @return a stream with the batch size updated
111     */
112    CacheStream<R> distributedBatchSize(int batchSize);
113 
114    /**
115     * Allows registration of a segment completion listener that is notified when a segment has completed
116     * processing.  If the terminal operator has a short circuit this listener may never be called.
117     * <p>This method is designed for the sole purpose of use with the {@link CacheStream#iterator()} to allow for
118     * a user to track completion of segments as they are returned from the iterator.  Behavior of other methods
119     * is not specified.  Please see {@link CacheStream#iterator()} for more information.</p>
120     * <p>Multiple listeners may be registered upon multiple invocations of this method.  The ordering of notified
121     * listeners is not specified.</p>
122     * @param listener The listener that will be called back as segments are completed.
123     * @return a stream with the listener registered.
124     */
125    CacheStream<R> segmentCompletionListener(SegmentCompletionListener listener);
126 
127    /**
128     * Disables tracking of rehash events that could occur to the underlying cache.  If a rehash event occurs while
129     * a terminal operation is being performed it is possible for some values that are in the cache to not be found.
130     * Note that you will never have an entry duplicated when rehash awareness is disabled, only lost values.
131     * <p>Most terminal operations will run faster with rehash awareness disabled even without a rehash occuring.
132     * However if a rehash occurs with this disabled be prepared to possibly receive only a subset of values.</p>
133     * @return a stream with rehash awareness disabled.
134     */
135    CacheStream<R> disableRehashAware();
136 
137    /**
138     * Sets a given time to wait for a remote operation to respond by.  This timeout does nothing if the terminal
139     * operation does not go remote.
140     * <p>If a timeout does occur then a {@link java.util.concurrent.TimeoutException} is thrown from the terminal
141     * operation invoking thread or on the next call to the {@link Iterator} or {@link Spliterator}.</p>
142     * <p>Note that if a rehash occurs this timeout value is reset for the subsequent retry if rehash aware is
143     * enabled.</p>
144     * @param timeout the maximum time to wait
145     * @param unit the time unit of the timeout argument
146     * @return a stream with the timeout set
147     */
148    CacheStream<R> timeout(long timeout, TimeUnit unit);
149 
150    /**
151     * Functional interface that is used as a callback when segments are completed.  Please see
152     * {@link CacheStream#segmentCompletionListener(SegmentCompletionListener)} for more details.
153     * @since 8.0
154     */
155    @FunctionalInterface
156    interface SegmentCompletionListener {
157       /**
158        * Method invoked when the segment has been found to be consumed properly by the terminal operation.
159        * @param segments The segments that were completed
160        */
161       void segmentCompleted(Set<Integer> segments);
162    }
163 
164    /**
165     * {@inheritDoc}
166     * <p>This operation is performed remotely on the node that is the primary owner for the key tied to the entry(s)
167     * in this stream.</p>
168     * <p>NOTE: This method while being rehash aware has the lowest consistency of all of the operators.  This
169     * operation will be performed on every entry at least once in the cluster, as long as the originator doesn't go
170     * down while it is being performed.  This is due to how the distributed action is performed.  Essentially the
171     * {@link CacheStream#distributedBatchSize} value controls how many elements are processed per node at a time
172     * when rehash is enabled. After those are complete the keys are sent to the originator to confirm that those were
173     * processed.  If that node goes down during/before the response those keys will be processed a second time.</p>
174     * <p>This method is ran distributed by default with a distributed backing cache.  However if you wish for this
175     * operation to run locally you can use the {@code stream().iterator().forEachRemaining(action)} for a single
176     * threaded variant.  If you
177     * wish to have a parallel variant you can use {@link java.util.stream.StreamSupport#stream(Spliterator, boolean)}
178     * passing in the spliterator from the stream.  In either case remember you <b>must</b> close the stream after
179     * you are done processing the iterator or spliterator..</p>
180     * @param action
181     */
182    @Override
183    void forEach(Consumer<? super R> action);
184 
185    /**
186     * {@inheritDoc}
187     * <p>Usage of this operator requires closing this stream after you are done with the iterator.  The preferred
188     * usage is to use a try with resource block on the stream.</p>
189     * <p>This method has special usage with the {@link org.infinispan.CacheStream.SegmentCompletionListener} in
190     * that as entries are retrieved from the next method it will complete segments.</p>
191     * <p>This method obeys the {@link CacheStream#distributedBatchSize(int)}.  Note that when using methods such as
192     * {@link CacheStream#flatMap(Function)} that you will have possibly more than 1 element mapped to a given key
193     * so this doesn't guarantee that many number of entries are returned per batch.</p>
194     * <p>Note that the {@link Iterator#remove()} method is only supported if no intermediate operations have been
195     * applied to the stream and this is not a stream created from a {@link Cache#values()} collection.</p>
196     * @return the element iterator for this stream
197     */
198    @Override
199    Iterator<R> iterator();
200 
201    /**
202     * {@inheritDoc}
203     * <p>Usage of this operator requires closing this stream after you are done with the spliterator.  The preferred
204     * usage is to use a try with resource block on the stream.</p>
205     * @return the element spliterator for this stream
206     */
207    @Override
208    Spliterator<R> spliterator();
209 
210    /**
211     * {@inheritDoc}
212     * <p>This operation is performed entirely on the local node irrespective of the backing cache.  This
213     * operation will act as an intermediate iterator operation requiring data be brought locally for proper behavior.
214     * Beware this means it will require having all entries of this cache into memory at one time.  This is described in
215     * more detail at {@link CacheStream}</p>
216     * <p>Any subsequent intermediate operations and the terminal operation are also performed locally.</p>
217     * @return the new stream
218     */
219    @Override
220    Stream<R> sorted();
221 
222    /**
223     * {@inheritDoc}
224     * <p>This operation is performed entirely on the local node irrespective of the backing cache.  This
225     * operation will act as an intermediate iterator operation requiring data be brought locally for proper behavior.
226     * Beware this means it will require having all entries of this cache into memory at one time.  This is described in
227     * more detail at {@link CacheStream}</p>
228     * <p>Any subsequent intermediate operations and the terminal operation are then performed locally.</p>
229     * @param comparator the comparator to be used for sorting the elements
230     * @return the new stream
231     */
232    @Override
233    Stream<R> sorted(Comparator<? super R> comparator);
234 
235    /**
236     * {@inheritDoc}
237     * <p>This intermediate operation will be performed both remotely and locally to reduce how many elements
238     * are sent back from each node.  More specifically this operation is applied remotely on each node to only return
239     * up to the <b>maxSize</b> value and then the aggregated results are limited once again on the local node.</p>
240     * <p>This operation will act as an intermediate iterator operation requiring data be brought locally for proper
241     * behavior.  This is described in more detail in the {@link CacheStream} documentation</p>
242     * <p>Any subsequent intermediate operations and the terminal operation are then performed locally.</p>
243     * @param maxSize how many elements to limit this stream to.
244     * @return the new stream
245     */
246    @Override
247    Stream<R> limit(long maxSize);
248 
249    /**
250     * {@inheritDoc}
251     * <p>This operation is performed entirely on the local node irrespective of the backing cache.  This
252     * operation will act as an intermediate iterator operation requiring data be brought locally for proper behavior.
253     * This is described in more detail in the {@link CacheStream} documentation</p>
254     * <p>Depending on the terminal operator this may or may not require all entries or a subset after skip is applied
255     * to be in memory all at once.</p>
256     * <p>Any subsequent intermediate operations and the terminal operation are then performed locally.</p>
257     * @param n how many elements to skip from this stream
258     * @return the new stream
259     */
260    @Override
261    Stream<R> skip(long n);
262 
263    /**
264     * {@inheritDoc}
265     * <p>This operation will be invoked both remotely and locally when used with a distributed cache backing this stream.
266     * This operation will act as an intermediate iterator operation requiring data be brought locally for proper
267     * behavior.  This is described in more detail in the {@link CacheStream} documentation</p>
268     * <p>This intermediate iterator operation will be performed locally and remotely requiring possibly a subset of
269     * all elements to be in memory</p>
270     * <p>Any subsequent intermediate operations and the terminal operation are then performed locally.</p>
271     * @return the new stream
272     */
273    @Override
274    Stream<R> distinct();
275 
276    /**
277     * {@inheritDoc}
278     * <p>Note when using a distributed backing cache for this stream the collector must be marshallable.  This
279     * prevents the usage of {@link java.util.stream.Collectors} class.  However you can use the
280     * {@link org.infinispan.stream.CacheCollectors} static factory methods to create a serializable wrapper, which then
281     * creates the actual collector lazily after being deserialized.  This is useful to use any method from the
282     * {@link java.util.stream.Collectors} class as you would normally.</p>
283     * @param collector
284     * @param <R1> collected type
285     * @param <A> intermediate collected type if applicable
286     * @return the collected value
287     * @see org.infinispan.stream.CacheCollectors
288     */
289    @Override
290    <R1, A> R1 collect(Collector<? super R, A, R1> collector);
